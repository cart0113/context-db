# context-md

A standard for organizing LLM context as a hierarchy of Markdown files with auto-generated tables of contents.

## Why

Many teams use [SKILL.md](https://github.com/anthropics/skills) for general project context because it's standard and portable. But skills are designed for *procedures* — filling them with background knowledge degrades skill performance and conflates "what to do" with "what to know."

context-md takes the good parts of skills (standard structure, portability via symlinks, front matter) and applies them specifically to background knowledge and project context. The hierarchical folder structure enables **progressive disclosure** — an LLM reads a lightweight index first, then fetches only the context relevant to the current task.

**Skills** → *what to do*
**context-md** → *what to know*

## How It Works

This project's own `CONTEXT/` folder is a working example:

```
CONTEXT/
├── context.cfg                          ← root config (plain YAML)
├── context_toc.md                       ← auto-generated root index (never edit)
└── context_md_project/                  ← subfolder (could be a symlink)
    ├── context_md_project.md            ← describes this subfolder (human-edited)
    ├── context_md_project_toc.md        ← auto-generated index
    ├── overview.md                      ← a context document
    └── design_decisions/
        ├── design_decisions.md
        ├── design_decisions_toc.md
        ├── bash_over_python.md
        ├── fenced_yaml_format.md
        ├── naming_convention.md
        └── no_cross_symlink_edits.md
```

The root `CONTEXT/` directory has a `context.cfg` (plain YAML) and a generated `context_toc.md`. Each subfolder has `<folder>.md` (human-edited) and `<folder>_toc.md` (generated). The LLM reads `_toc.md` files to discover what's available, then fetches individual documents as needed.

Subfolders can be **symlinks** to shared resources from other projects. The script reads across symlinks (to get descriptions) but never writes into them — they appear as *(read-only)* in the TOC.

### CONTEXT/context.cfg

The root config file. Plain YAML with at minimum a `description`:

```yaml
description: Context for the context-md project — the standard itself, design decisions, and tooling
```

Optional config fields can be added:

```yaml
description: Context for the context-md project
ignore: [scratch, old_docs]
follow_symlinks: [CODING]
```

### CONTEXT/context_toc.md (generated)

`build_toc.sh` produces this by reading `context.cfg` and scanning the directory:

```markdown
<!-- Auto-generated by build_toc.sh — edit context.cfg instead -->

# CONTEXT

> Context for the context-md project — the standard itself, design decisions, and tooling

## Subfolders

| Folder | Description |
| ------ | ----------- |
| [context_md_project/](context_md_project/context_md_project_toc.md) | The context-md standard — design decisions and tooling |

## Files

_No context files found._
```

### Subfolder description: context_md_project.md

Each subfolder has a `<folder>.md` file with a fenced YAML description block:

~~~markdown
```yaml description
description: The context-md standard — design decisions and tooling
```

Key constraint: `build_toc.sh` must stay bash 3.2 compatible (macOS system bash).
~~~

### Context documents

Individual `.md` files use standard YAML front matter:

```markdown
---
description: Why build_toc.sh is pure bash rather than Python or another language
---

(content)
```

The `description` field is what appears in the parent's `_toc.md`.

## Naming Convention

Subfolder files are named after their parent folder:

| Folder | Human file | Generated file |
|--------|------------|----------------|
| `context_md_project/` | `context_md_project.md` | `context_md_project_toc.md` |
| `design_decisions/` | `design_decisions.md` | `design_decisions_toc.md` |
| `CODING/` | `CODING.md` | `CODING_toc.md` |

The root is special: `context.cfg` + `context_toc.md`.

## Config

Root-level config goes in `context.cfg` (plain YAML). Subfolder config uses an optional fenced block in `<folder>.md`:

~~~markdown
```yaml config
ignore: [scratch, old_docs]
follow_symlinks: [CODING]
```
~~~

| Option | Default | Description |
|--------|---------|-------------|
| `ignore` | `[]` | Folders/files to exclude from the TOC and skip during walk |
| `follow_symlinks` | `[]` | Symlinked folders to recurse into (by default symlinks are read-only) |

Underscore-prefixed and dot-prefixed names are always skipped.

## Building

```bash
bin/build_toc.sh                    # Recursively build all _toc.md files
bin/build_toc.sh --check            # Only rebuild if sources changed
bin/build_toc.sh CONTEXT/           # Build from a specific directory
```

The script walks the directory tree recursively. It only writes `_toc.md` files whose real path is under the project root — so symlinks to external projects are never modified.

### Pre-commit hook

```bash
cp hooks/pre-commit .git/hooks/pre-commit
chmod +x .git/hooks/pre-commit
```

Uses `--check` mode so only changed files trigger a rebuild.

## Agent Entry Point

The LLM needs to know this system exists. The `bootstrap/` folder contains a skill and system instructions — wire one of these into your agent's startup:

**Skill** (Claude Code, Codex):
```bash
mkdir -p skills
ln -s /path/to/context-md/bootstrap/skill.md skills/context-md.md
```

**Cursor rule:**
```bash
mkdir -p .cursor/rules
cp /path/to/context-md/templates/cursor-rule.mdc .cursor/rules/context-md.mdc
```

## Progressive Disclosure

The folder hierarchy is designed for progressive disclosure:

1. **LLM reads `context_toc.md`** — sees one-line descriptions of every subfolder and file
2. **Decides what's relevant** — a database question doesn't need coding style docs loaded
3. **Fetches only what it needs** — keeps the context window focused
4. **For subfolders, reads their `_toc.md` first** — same pattern repeats at each level

This keeps token usage low and prevents context pollution. Background knowledge is organized the same way a human would organize documentation — by topic, with summaries at each level.
