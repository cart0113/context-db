# context-md

Organize LLM context as modular, auto-discovered Markdown files.

## Example

A project using context-md:

```
my_project/
├── src/
└── CONTEXT/
    ├── CONTEXT.md                  ← describes this context node
    ├── CONTEXT_toc.md              ← auto-generated index (never edit)
    ├── architecture.md             ← a context document
    ├── CODING/ → /shared/coding    ← symlink to shared coding standards
    │   ├── CODING.md
    │   ├── CODING_toc.md
    │   └── defensive.md
    └── DATABASE/
        ├── DATABASE.md
        ├── DATABASE_toc.md
        └── schema.md
```

### CONTEXT/CONTEXT.md

The human-edited file. Describes what this context node is about:

~~~markdown
```yaml description
title: My Project
description: Payments service — architecture, coding style, and database conventions
```

```yaml config
read_only: [CODING]
```

When writing or reviewing code, load CODING/.
For database work, load DATABASE/.
~~~

The `description` block is required. The `config` block is optional — most context
nodes only need `description`.

### CONTEXT/CONTEXT_toc.md (generated)

`build_toc.sh` produces this from `CONTEXT.md` configs and front matter of resources:

```markdown
<!-- Auto-generated by build_toc.sh — edit CONTEXT.md instead -->

# My Project

> Payments service — architecture, coding style, and database conventions

## Subfolders

| Folder | Description |
| ------ | ----------- |
| [CODING/](CODING/CODING_toc.md) | Shared team coding standards *(read-only)* |
| [DATABASE/](DATABASE/DATABASE_toc.md) | Database schema and migration patterns |

## Files

| File | Description |
| ---- | ----------- |
| [architecture.md](architecture.md) | System components and data flow |
```

### Context documents

Each `.md` file has standard YAML front matter:

```markdown
---
title: Architecture Overview
description: System components and data flow
---

(content)
```

## Naming Convention

Files follow the folder name:

| Folder | Human file | Generated file |
|--------|------------|----------------|
| `CONTEXT/` | `CONTEXT.md` | `CONTEXT_toc.md` |
| `CODING/` | `CODING.md` | `CODING_toc.md` |
| `DATABASE/` | `DATABASE.md` | `DATABASE_toc.md` |

## Config Options

Set these in the `` ```yaml config `` block (all optional):

| Option | Default | Description |
|--------|---------|-------------|
| `skip_underscore` | `true` | Skip files/folders starting with `_` |
| `ignore` | `[]` | Names to exclude from the TOC |
| `read_only` | `[]` | Include in TOC but don't rebuild (for symlinks) |
| `eager_read` | `[]` | Folders the LLM should always load |

## Building

```bash
bin/build_toc.sh                    # build all
bin/build_toc.sh CONTEXT/           # build one directory
```

## Agent Entry Point

The LLM needs to know this system exists. The `bootstrap/` folder contains:

- **`bootstrap/skill.md`** — a skill file that instructs agents to read `CONTEXT_toc.md`
  files. Copy or symlink this into your project's skills directory.
- **`bootstrap/CONTEXT_MD_SYSTEM_INSTRUCTIONS.md`** — explains the context-md system
  to an LLM. Referenced by the skill; can also be inserted into a cursor rule, system
  prompt, or any other agent entry point.

```bash
# Claude Code / Codex — symlink the skill
ln -s /path/to/context-md/bootstrap/skill.md your_project/skills/context-md.md

# Cursor — copy the rule template
cp templates/cursor-rule.mdc your_project/.cursor/rules/context-md.mdc
```

## Why?

Many teams use [SKILL.md](https://github.com/anthropics/skills) for general context
because it's standard and portable via symlinks. But skills are designed for
*procedures* — filling them with background knowledge degrades skill performance and
creates confusion between "what to do" and "what to know."

context-md takes the good ideas from skills (standard structure, portability, front
matter) and applies them specifically to background knowledge and project context.

**Skills** → *what to do*
**context-md** → *what to know*

[Full rationale](https://cart0113.github.io/context-md/#/blog/why-context-md)

## Tools

| Script | Description |
|--------|-------------|
| `bin/build_toc.sh` | Rebuild `*_toc.md` files; run from project root or pass a path |
| `bin/format_md.py` | Format Markdown tables to fixed-width alignment |
| `hooks/pre-commit` | Git pre-commit hook that runs `build_toc.sh` |
