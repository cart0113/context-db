# Specification

## Overview

A context-md context node is a directory named `CONTEXT/` that contains three reserved
files and any number of context documents.

```
CONTEXT/
├── CONTEXT.yml              (required) config and metadata
├── CONTEXT_INSTRUCTIONS.md  (optional) LLM prose guidance
├── CONTEXT_TOC.md           (generated) full discoverable index
├── SYSTEM.md                (optional) one-time system explanation
└── *.md                     (optional) context documents
```

## Reserved Files

### CONTEXT.yml

The config file for this context node. Required for `build_toc.sh` to process the
directory. Defines the node's identity (title, description) and controls what the
script indexes.

```yaml
title: Human-readable name for this context node
description: One-line summary (used in parent's Subfolders table)

depth: 1
skip_underscore: true
ignore: []
read_only: []
eager_read: []
```

**Config fields:**

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `title` | string | `(untitled)` | Heading in the generated CONTEXT_TOC.md |
| `description` | string | *(empty)* | Summary shown in parent TOC entries |
| `depth` | integer | `1` | Discovery depth; currently only depth=1 is implemented |
| `skip_underscore` | boolean | `true` | Skip files/folders beginning with `_` |
| `ignore` | list | `[]` | Names (without trailing `/`) to exclude entirely |
| `read_only` | list | `[]` | Names to include in TOC but not rebuild (for external/symlinked resources) |
| `eager_read` | list | `[]` | Folder names the LLM should always load alongside this TOC |

### CONTEXT_INSTRUCTIONS.md

Optional Markdown prose addressed to LLMs. The full content is included verbatim in
the generated `CONTEXT_TOC.md` under an "Instructions" heading. Use this to give
context-specific guidance: which subfolder to load for what kind of task, key
relationships to be aware of, caveats, etc.

Human-edited. Never modified by the script.

### CONTEXT_TOC.md

Fully generated by `build_toc.sh`. Do not edit by hand. The script regenerates the
entire file on each run. See [TOC Format](toc-format.md) for the output structure.

### SYSTEM.md

Optional one-time system explanation. Explains the context-md system to an LLM that
has never encountered it. The script skips this file when building the Files table —
it is never indexed and never appears in any TOC. It exists only to be read directly.

Place one `SYSTEM.md` at the top-level `CONTEXT/` of your project. If a project
includes another project's context via symlink, that linked project's `SYSTEM.md` does
not need to be read again.

## Context Documents

Any `.md` file in a `CONTEXT/` directory (other than the reserved files above) is a
context document. Each should have YAML front matter:

```markdown
---
title: Document Title
description: One-line summary for the TOC entry
---

# Document Title

Content...
```

- `title` — used as the link label in the TOC Files table
- `description` — the description column in the TOC Files table
- If either field is missing, the script falls back to the filename (without `.md`)

## Discovery Rules

When `build_toc.sh` processes a `CONTEXT.yml`, it builds the Subfolders and Files
tables by scanning two locations:

**Subfolders table** — populated from:
1. Immediate subdirectories of `CONTEXT/` that contain a `CONTEXT.yml`
2. Directories adjacent to the project root (siblings of the `CONTEXT/` parent) that
   contain a `CONTEXT/CONTEXT.yml`

**Files table** — populated from:
- `.md` files directly in `CONTEXT/` (reserved filenames are always excluded)

**Filtering:**
- `skip_underscore: true` (default) — silently skips anything starting with `_`
- `ignore` list — silently skips listed names
- `read_only` list — includes the name in the TOC but marks it *(read-only)* and skips
  rebuilding its own `CONTEXT_TOC.md` when running in auto-discovery mode

## Portability via Symlinks

Any `CONTEXT/` subfolder can be a symlink to an external directory. The discovery
rules follow symlinks normally. To prevent `build_toc.sh` from modifying files inside
a symlinked external resource, add its name to the `read_only` list.

```yaml
read_only: [SHARED_STYLE]  # symlinked from another repo — don't rebuild
```
