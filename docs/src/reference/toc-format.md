# TOC Format

`<folder>-toc.md` files are generated by `build_toc.sh`. Never edit them by
hand.

## Example

From the acme-payments example (`acme-payments-toc.md`):

<!-- prettier-ignore -->
```markdown
## Subfolders

- description: Database schema, entities, and relationships
  path: data-model/data-model-toc.md

## Files

- description: REST API endpoints, authentication, and error codes
  path: api-reference.md
- description: System components, data flow, and service boundaries
  path: architecture.md
```

Each entry has `description:` on the first line and `path:` on the second.

## Sections

| Section    | Contents                                                       |
| ---------- | -------------------------------------------------------------- |
| Subfolders | Subdirectories that have a description file                    |
| Files      | `.md` files (excluding the description file and the `-toc.md`) |

Sections only appear when there are entries. An empty folder produces an empty
TOC. Underscore-prefixed and dot-prefixed names are excluded.

## How agents use TOCs

An agent navigating a context-db reads TOCs top-down:

```
context-db-toc.md
  → sees "Acme Payments" subfolder → reads acme-payments-toc.md
    → sees "Data Model" subfolder → reads data-model-toc.md
      → sees "entities.md" → fetches it
```

At each level, the agent decides what to fetch based on descriptions. This keeps
context loading proportional to task scope rather than total knowledge size.
