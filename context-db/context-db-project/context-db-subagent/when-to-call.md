# When to Call the Subagent

## The problem

Calling the subagent on every prompt and every code change is wasteful. Each
call takes 10-30 seconds and costs tokens. A follow-up like "make that variable
name shorter" does not need a fresh knowledge base lookup or convention review.

The challenge is teaching the main agent to judge when a call is worth the cost.

## pre-review — immediately before making edits

Pre-review runs as step 1 of the combined workflow, before any code changes. The
agent sends a structured plan and gets back verbatim applicable standards.

What should NOT trigger pre-review:

- Pure git operations (commit, push, branch).
- Explanation requests with no edits planned.
- Follow-up refinements on the same task (standards already fetched).

What SHOULD trigger pre-review:

- Any task that involves writing or editing code, docs, or config.
- Starting a new task of a different type than the previous one.
- Anything where coding or writing standards are relevant.

## user-prompt — before starting work

The `when` config controls frequency: `always`, `major`, or `never`.

What should NOT trigger a user-prompt call:

- Follow-up questions on the same topic ("make it async instead").
- Small refinements ("rename that variable").
- Git operations after coding ("commit this").
- Requests for explanation of code already written.
- Yes/no confirmations.

What SHOULD trigger a user-prompt call:

- New feature work or bug fixes.
- Switching to a different area of the codebase.
- Documentation or test writing (different conventions may apply).
- Anything where the developer is starting fresh on a topic.

## code-review — after code changes

Code-review runs AFTER the main agent has made code changes. The flow:

1. Before making changes, ensure the repo is committed (clean baseline). If
   uncommitted changes exist, ask the user whether to commit.
2. Make the code changes.
3. Run the code-review script — it calls `git diff` itself.
4. Read the review response. It may have been generated by a less capable model
   — treat it as advisory.
5. Fix real issues, ignore false positives.

What should NOT trigger a code-review:

- Single-line fixes.
- Variable renames or formatting.
- Adding a log statement.
- Changes the main agent is confident about from a prior user-prompt.

What SHOULD trigger a code-review:

- New files or significant new functions.
- Changes to API contracts, schemas, or public interfaces.
- Anything touching areas flagged in the user-prompt response.
- Architectural changes or new patterns.

### Review type

Configurable via `review-type` in `.contextdb.json`:

- **context-db** (default, sonnet): only flag issues that are backed by a
  convention in the knowledge base. Each critique cites the context-db source.
- **full** (opus): also do a general code review based on the model's own
  expertise. The report has two clearly separated sections.

## update-context-db — after completing work

The subagent receives the developer's notes, runs `git diff` to see what code
changed, navigates context-db, and makes edits/writes directly.

What should NOT trigger an update:

- Routine changes with no surprises.
- Work that followed conventions perfectly.
- Tasks with no user corrections.

What SHOULD trigger an update:

- The user corrected the agent's approach.
- A convention was discovered that wasn't in the knowledge base.
- A pitfall was hit that would catch the next agent.
- Something surprising about the codebase was learned.
