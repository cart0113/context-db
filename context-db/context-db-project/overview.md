# context-db — Overview

## What It Is

A standard for organizing project knowledge as hierarchical Markdown files with auto-generated tables of contents. An LLM reads a lightweight TOC, sees one-line descriptions of every file and subfolder, and fetches only what's relevant to its current task.

## Why It Was Built

Large projects need more background knowledge than a single context file can hold. But dumping everything into the LLM's context window degrades performance — research shows accuracy drops as input grows, especially with irrelevant content.

The fix is smaller documents organized in folders. But then agents need a way to discover what exists without loading everything. context-db solves this with auto-generated `-toc.md` indexes: lightweight maps that show the agent what knowledge is available, so it can fetch selectively.

## Core Design Principles

1. **Descriptions are the interface.** The `description` in each file's frontmatter is the only thing an agent sees before deciding to read it. Good descriptions keep the agent focused; bad ones waste tokens or cause missed context.

2. **Structure is retrieval.** The folder hierarchy isn't just organization — each folder becomes a TOC node. Deeper nesting means more granular filtering before committing tokens.

3. **TOCs are derived.** The agent maintains content and descriptions. The indexes are generated by `bin/build_toc.sh` — never hand-edited.

## What It Is Not

Not a replacement for procedural tools like SKILL.md or workflows. context-db is for *knowledge* — architecture, conventions, domain context, design decisions — not step-by-step instructions.
